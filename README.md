# yadro-test-tatlin-object-core-team
РЕАЛИЗОВАНО В РАМКАХ ТЕСТОВОГО ЗАДАНИЯ ДЛЯ КОМПАНИИ YADRO КОМАНДЫ TATLIN.OBJECT CORE

### Описание
Приложение получает на вход имя файла, считает количество вхождений каждого имени в файле и выводит результат в STDOUT.
Пример работы:

IN(test.txt):
```txt
Даша
Даша
Маша
Маша
Рома
```

OUT:
```zsh
Даша:2
Рома:1
Маша:2
```

### Особенность реализации
Особенность реализации заключалась в вопросах, поставленных в начале разработки:

1. Что делать если файл на вход будет слишком большим?
   
  Ответ: Может случится out of memory, так же обработка данных будет крайне долгой.

  Реализация решения проблемы: приложение работает в виде streaming pipeline, то есть мы подгружаем данные из файла построчно по частям. Далее данные передаются в каналы бакетов. Реализована hash-based sharding система, когда имена по хэшу кладутся в отдельные бакет файлы, которые в конечном итоге считываются в local map для каждого бакета, затем в global result map для конечного результата. Это позволило не грузить в память сразу весь огромный файл, а работать с небольшими частями файла, распределяя их по хэшу в отдельные файлы, суммируя по отдельности результат. 

2. Какое тестирование мы можем провести?

  Ответ: Можем провести unit, integration, concurrency тестирование для различных компонентов системы.

  Реализация: Было произведено как unit тестирование, так и integration, что позволило проверить работоспособность как отдельных частей системы, так и работу пайплайна целиком. Также были небольшие concurrency кейсы, где проверилась корректная работа с context, синхронизация через wait group и каналы, отсутствие гонок данных при помощи -race. Общее покрытие тестами: ~85%

3. Можем ли мы выводить имена в порядке убывания частоты?

  Ответ: Да, можем. Однако данный вопрос не был реализован в рамках приложения, потому что кейс является не обязательным с точки зрения корректной работоспособности приложения, а также требует дополнительной памяти и сложности ради визуального представления результата

  Реализация: в дальнейшем после reduce стадии пайплайна, мы можем сложить в слайс структур Pair {key, value} значения result map, отсортировать и вывести в порядке убывания.

### Конфигурация
Приложение конфигурируется при помощи файла _config.yml_, в котором можно задать размер batch, channel buffer и bucket count. Пример файла:
```yaml
chan_buffer: 1000
batch_size: 500
bucket_count: 5
```

Также приложение содержит статический анализатор GolangCI-lint, который также конфигурируется при помощи файла _.golangci.yml_. Пример файла:
```yaml
version: "2"

run:
  timeout: 5m

linters:
  enable:
    - govet
    - ineffassign
    - errcheck

    - staticcheck
    - unused

    - bodyclose
    - noctx

    - prealloc
formatters:
  enable:
    - gofmt
    - goimports
```

### Работа с приложением
Для удобной работы с приложением были описаны таргеты Makefile файла:
| Command | Description |
|---|---|
| `make build` | Build application binary |
| `make run` | Run application with default input file |
| `make unit_test` | Run unit tests |
| `make integration_test` | Run integration tests with race detector -race|
| `make lint` | Run linters |
| `make clean` | Remove build artifacts and temporary files |
| `make cover` | Get coverage of all tests |

Также сам файл содержит удобные константы, которые можно изменять под свои нужды: название бинарника(**BIN_NAME**), путь к конфиг файлу(**CONFIG_FILE**).
